## AuthnRequest `IssueInstant` is required

Inbound `AuthnRequest` messages must now include an unqualified `IssueInstant`
with a valid UTC `xs:dateTime` value ending in `Z`. The check applies to HTTP
POST, HTTP-Redirect, and HTTP-POST-SimpleSign.

### Who needs to change

No source change is required when requests are generated through the standard
`saml-rs` APIs, which already emit `IssueInstant`. Changes are required for:

- custom `AuthnRequest` XML or templates;
- tests that manually construct `FlowResult` extraction data;
- integrations whose upstream service provider omits the attribute or sends a
  local-offset timestamp.

### Update custom request XML

Before, a custom request without `IssueInstant` could reach the normal request
flow:

```xml
<samlp:AuthnRequest
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    ID="_request"
    Version="2.0"/>
```

After upgrading, include a UTC timestamp ending in `Z`:

```xml
<samlp:AuthnRequest
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    ID="_request"
    Version="2.0"
    IssueInstant="2026-07-16T23:13:10Z"/>
```

Fractional seconds are accepted, but offsets such as `+00:00` are not a
substitute for the required `Z` form. The attribute itself must be unqualified,
even when the request element uses a namespace prefix.

### Update manually constructed raw fixtures

Tests or compatibility code that constructs `FlowResult` directly must add
`request.issueInstant` to the extracted request object.

Before:

```json
{
  "request": {
    "id": "_request"
  }
}
```

After:

```json
{
  "request": {
    "id": "_request",
    "issueInstant": "2026-07-16T23:13:10Z"
  }
}
```

Typed handlers can read the whitespace-normalized value through the additive
accessor:

```rust
let issue_instant = request.issue_instant().as_str();
```

### Validation behavior

Missing, malformed, or non-UTC values now return
`SamlError::ProtocolProfile` before signature handling. Surrounding XML Schema
whitespace is normalized before the value is exposed through
`AuthnRequest::issue_instant()`.

Valid older timestamps remain accepted. This change validates presence and
format without adding a freshness window, clock-skew setting, or replay policy.

## LogoutResponse `IssueInstant` is required

As of [PR #80](https://github.com/salasebas/opensaml-rs/pull/80), inbound
`LogoutResponse` messages must include an unqualified `IssueInstant` with a
valid SAML UTC `xs:dateTime` value ending in `Z`. The check applies to HTTP
POST, HTTP-Redirect, and HTTP-POST-SimpleSign.

The low-level raw flow, both public raw parsers (`parse_logout_response` and
`parse_logout_response_without_request_id`), typed conversion from
`FlowResult`, and typed `finish_slo` handling all enforce the same rule.

### Who needs to change for LogoutResponse

No source change is required for conforming messages generated through the
standard `saml-rs` APIs, which already emit `IssueInstant`. Changes are
required for:

- custom `LogoutResponse` XML or templates that omit the attribute;
- tests or compatibility code that manually construct `FlowResult` extraction
  data without `response.issueInstant`;
- integrations whose upstream identity provider omits the attribute or sends
  a local-offset timestamp.

### Update custom LogoutResponse XML

Before, a custom response without `IssueInstant` could reach the normal
response flow:

```xml
<samlp:LogoutResponse
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    ID="_response"
    Version="2.0">
    <samlp:Status>
        <samlp:StatusCode
            Value="urn:oasis:names:tc:SAML:2.0:status:Success"/>
    </samlp:Status>
</samlp:LogoutResponse>
```

After upgrading, include a UTC timestamp ending in `Z`:

```xml
<samlp:LogoutResponse
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    ID="_response"
    Version="2.0"
    IssueInstant="2026-07-16T23:13:10Z">
    <samlp:Status>
        <samlp:StatusCode
            Value="urn:oasis:names:tc:SAML:2.0:status:Success"/>
    </samlp:Status>
</samlp:LogoutResponse>
```

Fractional seconds are accepted, but offsets such as `+00:00` are not a
substitute for the required `Z` form. The attribute itself must be unqualified;
for example, `samlp:IssueInstant` is rejected.

### Update manually constructed LogoutResponse fixtures

Tests or compatibility code that constructs `FlowResult` directly must add
`response.issueInstant` to the extracted response object.

Before:

```json
{
  "response": {
    "id": "_response"
  }
}
```

After:

```json
{
  "response": {
    "id": "_response",
    "issueInstant": "2026-07-16T23:13:10Z"
  }
}
```

### Read the normalized timestamp

Typed handlers can read the XML Schema whitespace-normalized value through the
additive accessor:

```rust
let issue_instant = response.issue_instant().as_str();
```

### LogoutResponse validation behavior

Missing, namespace-qualified, malformed, or non-UTC values return
`SamlError::ProtocolProfile` before signature handling.

SAML Core 2.0 section 1.3.3 says producers `MUST NOT` generate timestamps that
contain leap seconds. `saml-rs` rejects rather than normalizes that
non-conformant input as a strict receiver policy; this is crate policy, not a
separately stated receiver-side `MUST` in the specification.

Valid older UTC timestamps remain accepted. This change adds no maximum message
age, freshness window, clock-skew policy, or replay default. In particular,
callers of the uncorrelated raw parser
`parse_logout_response_without_request_id` still own request correlation,
replay protection, and any optional freshness policy.

## Migration checklist

- Add a UTC `IssueInstant` ending in `Z` to every custom `AuthnRequest`.
- Add `request.issueInstant` to manually constructed `FlowResult` fixtures.
- Add a UTC `IssueInstant` ending in `Z` to every custom `LogoutResponse`.
- Add `response.issueInstant` to manually constructed `FlowResult` fixtures.
- Update tests or telemetry that expect another error for malformed unsigned
  requests.
- No change is needed for requests generated by the standard `saml-rs` APIs.
