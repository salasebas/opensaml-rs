# Migrating from 0.3 to 0.4

This guide covers the changes consumers need to make when upgrading from
`0.3` to `0.4`. For the complete list of changes, see the
[changelog](../../CHANGELOG.md).

## AuthnRequest `IssueInstant` is required

Inbound `AuthnRequest` messages must now include an unqualified `IssueInstant`
with a valid UTC `xs:dateTime` value ending in `Z`. The check applies to HTTP
POST, HTTP-Redirect, and HTTP-POST-SimpleSign.

### Who needs to change

No source change is required when requests are generated through the standard
`saml-rs` APIs, which already emit `IssueInstant`. Changes are required for:

- custom `AuthnRequest` XML or templates;
- tests that manually construct `FlowResult` extraction data;
- integrations whose upstream service provider omits the attribute or sends a
  local-offset timestamp.

### Update custom request XML

Before, a custom request without `IssueInstant` could reach the normal request
flow:

```xml
<samlp:AuthnRequest
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    ID="_request"
    Version="2.0"/>
```

After upgrading, include a UTC timestamp ending in `Z`:

```xml
<samlp:AuthnRequest
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    ID="_request"
    Version="2.0"
    IssueInstant="2026-07-16T23:13:10Z"/>
```

Fractional seconds are accepted, but offsets such as `+00:00` are not a
substitute for the required `Z` form. The attribute itself must be unqualified,
even when the request element uses a namespace prefix.

### Update manually constructed raw fixtures

Tests or compatibility code that constructs `FlowResult` directly must add
`request.issueInstant` to the extracted request object.

Before:

```json
{
  "request": {
    "id": "_request"
  }
}
```

After:

```json
{
  "request": {
    "id": "_request",
    "issueInstant": "2026-07-16T23:13:10Z"
  }
}
```

Typed handlers can read the whitespace-normalized value through the additive
accessor:

```rust
let issue_instant = request.issue_instant().as_str();
```

### Validation behavior

Missing, malformed, or non-UTC values now return
`SamlError::ProtocolProfile` before signature handling. Surrounding XML Schema
whitespace is normalized before the value is exposed through
`AuthnRequest::issue_instant()`.

Valid older timestamps remain accepted. This change validates presence and
format without adding a freshness window, clock-skew setting, or replay policy.

## SSO Response and Assertion `IssueInstant` values are required

As of [PR #90](https://github.com/salasebas/opensaml-rs/pull/90), an inbound
SSO `Response` and its selected `Assertion` must each include an unqualified
`IssueInstant` with a valid SAML UTC `xs:dateTime` value ending in `Z`.

The rule applies to raw POST, Redirect, and SimpleSign parsing; conversion to
`SsoResponse` or `SsoSession`; solicited `finish_sso`; and unsolicited SSO on
the bindings supported by those typed flows. Selected assertions recovered
through XML Encryption are revalidated after decryption.

This is a behavioral breaking change rather than a public API removal or
rename. The typed timestamp accessors described below are additive.

### Who needs to change for SSO

No source change is required for conforming responses generated through the
standard `saml-rs` APIs, which already emit both attributes. Changes are
required for:

- custom SSO `Response` or `Assertion` XML that omits either attribute;
- tests or compatibility code that manually construct `FlowResult` extraction
  data without `response.issueInstant` or `assertion.issueInstant`;
- integrations whose upstream identity provider sends a namespace-qualified
  attribute, a local-offset timestamp, or another malformed value.

### Update custom SSO response XML

Before, custom XML could omit the timestamps from either level:

```xml
<samlp:Response
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    xmlns:saml="urn:oasis:names:tc:SAML:2.0:assertion"
    ID="_response"
    Version="2.0">
    <!-- Status and other response content -->
    <saml:Assertion ID="_assertion" Version="2.0">
        <!-- Assertion content -->
    </saml:Assertion>
</samlp:Response>
```

After upgrading, include an unqualified UTC timestamp on both elements:

```xml
<samlp:Response
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    xmlns:saml="urn:oasis:names:tc:SAML:2.0:assertion"
    ID="_response"
    Version="2.0"
    IssueInstant="2026-07-20T16:00:00Z">
    <!-- Status and other response content -->
    <saml:Assertion
        ID="_assertion"
        Version="2.0"
        IssueInstant="2026-07-20T16:00:01Z">
        <!-- Assertion content -->
    </saml:Assertion>
</samlp:Response>
```

Fractional seconds are accepted. Offsets such as `+00:00` are not substitutes
for the required `Z` form. A qualified attribute such as
`saml:IssueInstant` does not satisfy the required unqualified attribute.

### Update manually constructed SSO fixtures

Code that converts a manually constructed `FlowResult` into `SsoResponse` must
provide `response.issueInstant`. Conversion into `SsoSession` requires both
the response and selected-assertion values.

Before:

```json
{
  "response": {
    "id": "_response"
  },
  "assertion": {
    "id": "_assertion"
  }
}
```

After:

```json
{
  "response": {
    "id": "_response",
    "issueInstant": "2026-07-20T16:00:00Z"
  },
  "assertion": {
    "id": "_assertion",
    "issueInstant": "2026-07-20T16:00:01Z"
  }
}
```

### Read the normalized SSO timestamps

The additive typed accessors expose the XML Schema whitespace-normalized
values without exposing the crate's internal date-time type:

```rust
let envelope_instant = response.issue_instant().as_str();
let response_instant = session.response_issue_instant().as_str();
let assertion_instant = session.assertion_issue_instant().as_str();
```

### SSO validation behavior

Missing, namespace-qualified, malformed, or non-UTC values return
`SamlError::ProtocolProfile`. Decrypted assertions are subject to the same
profile validation before the implementation selects trusted signature
coverage.

Valid older UTC timestamps remain accepted. Presence and lexical validation do
not add a maximum message age, freshness window, clock-skew policy, or replay
default.

## LogoutResponse `IssueInstant` is required

As of [PR #80](https://github.com/salasebas/opensaml-rs/pull/80), inbound
`LogoutResponse` messages must include an unqualified `IssueInstant` with a
valid SAML UTC `xs:dateTime` value ending in `Z`. The check applies to HTTP
POST, HTTP-Redirect, and HTTP-POST-SimpleSign.

The low-level raw flow, both public raw parsers (`parse_logout_response` and
`parse_logout_response_without_request_id`), typed conversion from
`FlowResult`, and typed `finish_slo` handling all enforce the same rule.

### Who needs to change for LogoutResponse

No source change is required for conforming messages generated through the
standard `saml-rs` APIs, which already emit `IssueInstant`. Changes are
required for:

- custom `LogoutResponse` XML or templates that omit the attribute;
- tests or compatibility code that manually construct `FlowResult` extraction
  data without `response.issueInstant`;
- integrations whose upstream identity provider omits the attribute or sends
  a local-offset timestamp.

### Update custom LogoutResponse XML

Before, a custom response without `IssueInstant` could reach the normal
response flow:

```xml
<samlp:LogoutResponse
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    ID="_response"
    Version="2.0">
    <samlp:Status>
        <samlp:StatusCode
            Value="urn:oasis:names:tc:SAML:2.0:status:Success"/>
    </samlp:Status>
</samlp:LogoutResponse>
```

After upgrading, include a UTC timestamp ending in `Z`:

```xml
<samlp:LogoutResponse
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    xmlns:saml="urn:oasis:names:tc:SAML:2.0:assertion"
    ID="{ID}"
    Version="2.0"
    IssueInstant="{IssueInstant}"
    Destination="{Destination}"
    InResponseTo="{InResponseTo}">
    <saml:Issuer>{Issuer}</saml:Issuer>
    <samlp:Status>
        <samlp:StatusCode
            Value="{StatusCode}"/>
    </samlp:Status>
</samlp:LogoutResponse>
```

Fractional seconds are accepted, but offsets such as `+00:00` are not a
substitute for the required `Z` form. The attribute itself must be unqualified;
for example, `samlp:IssueInstant` is rejected. The placeholders above also
satisfy the outbound template requirements described later in this guide.

### Update manually constructed LogoutResponse fixtures

Tests or compatibility code that constructs `FlowResult` directly must add
`response.issueInstant` to the extracted response object.

Before:

```json
{
  "response": {
    "id": "_response"
  }
}
```

After:

```json
{
  "response": {
    "id": "_response",
    "issueInstant": "2026-07-16T23:13:10Z"
  }
}
```

### Read the normalized timestamp

Typed handlers can read the XML Schema whitespace-normalized value through the
additive accessor:

```rust
let issue_instant = response.issue_instant().as_str();
```

### LogoutResponse validation behavior

Missing, namespace-qualified, malformed, or non-UTC values return
`SamlError::ProtocolProfile` before signature handling.

Valid older UTC timestamps remain accepted. This change adds no maximum message
age, freshness window, clock-skew policy, or replay default. In particular,
callers of the uncorrelated raw parser
`parse_logout_response_without_request_id` still own request correlation,
replay protection, and any optional freshness policy.

## Custom LogoutResponse templates are validated before dispatch

As of [PR #91](https://github.com/salasebas/opensaml-rs/pull/91), the final XML
rendered from a custom `logout_response_template` is validated before encoding
or signing. Signed HTTP-POST output is validated again after signing.

This is outbound producer validation. It does not make inbound parsing reject
additional peer representations unless a separate receiver or schema rule
already requires that behavior.

### Who needs to change custom LogoutResponse templates

No source change is required for applications that use the standard
`saml-rs` LogoutResponse renderer and do not call the removed
`RespondSlo::signing` method.

Review applications that:

- configure `TemplatePolicy::logout_response_template`;
- add custom attributes or SAML-defined children to a LogoutResponse;
- omit `Issuer`, `Status`, `Destination`, or `InResponseTo` from generated
  responses;
- provide a template-owned root `ds:Signature`;
- call `RespondSlo::signing(...)` to override typed response signing.

### Update custom LogoutResponse templates

The rendered root must be a SAML protocol `LogoutResponse` with the generated
`ID`, SAML version `2.0`, and a valid UTC `IssueInstant`. When a response
answers a known request, `InResponseTo` must be present, use the XML Schema
`NCName` lexical form, and match that request. Signed responses must include
the selected SLO endpoint as `Destination`.

The direct children must use this order:

1. required SAML assertion `Issuer`;
2. optional library-generated `ds:Signature`;
3. optional protocol `Extensions`;
4. exactly one final protocol `Status`.

The `Issuer` must identify the responding entity. If its `Format` is present,
it must be `urn:oasis:names:tc:SAML:2.0:nameid-format:entity`; its value must
be an absolute URI no longer than 1024 characters. Entity-format qualifiers
are not allowed.

`Extensions` must contain at least one namespace-qualified, non-SAML extension
element. Do not place SAML assertion or protocol elements directly inside it.
`Status` must contain exactly one leading `StatusCode`, followed only by an
optional `StatusMessage` and optional `StatusDetail`.

Use the built-in placeholders so the final XML remains correlated with the
generated message:

```xml
<samlp:LogoutResponse
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    xmlns:saml="urn:oasis:names:tc:SAML:2.0:assertion"
    ID="{ID}"
    Version="2.0"
    IssueInstant="{IssueInstant}"
    Destination="{Destination}"
    InResponseTo="{InResponseTo}">
    <saml:Issuer>{Issuer}</saml:Issuer>
    <samlp:Status>
        <samlp:StatusCode Value="{StatusCode}"/>
    </samlp:Status>
</samlp:LogoutResponse>
```

Do not include a root `ds:Signature` in the template. `saml-rs` constructs the
embedded HTTP-POST signature or detached HTTP-Redirect/SimpleSign signature
after validating the unsigned XML.

### Typed LogoutResponse output is always signed

`RespondSlo::signing` has been removed. Typed SLO response creation now always
signs front-channel LogoutResponses, even when the local validation policy is
configured for compatibility.

Before:

```rust
let response = idp.respond_slo(
    &sp,
    &request,
    RespondSlo::post().signing(LogoutSigning::DoNotSignForCompatibility),
)?;
```

After:

```rust
let response = idp.respond_slo(&sp, &request, RespondSlo::post())?;
```

Ensure the local typed configuration contains a usable signing key and
certificate. If an integration intentionally needs an unsigned,
non-conformant compatibility response, use the lower-level
`saml_rs::raw::logout::create_logout_response` API with `want_signed = false`
and keep that exception outside the typed conformant flow.

`LogoutSigning` remains available for `StartSlo`, so request-signing call sites
do not need to change solely because response signing became mandatory.

### Outbound template validation behavior

Malformed structure and lexical values return `SamlError::ProtocolProfile`.
Generated ID mismatches return `SamlError::Invalid`; issuer, destination, and
request-correlation mismatches retain their dedicated semantic error variants.
Failures occur before the message is dispatched.

Existing custom templates that relied on undeclared attributes, invalid child
ordering, SAML content inside `Extensions`, incomplete `Status`, or a
template-owned XML signature must be updated rather than relaxed through a
typed compatibility policy.

## Leap seconds are accepted inbound but never generated

XML Schema permits the lexical seconds value `60`. SAML Core 2.0 section 1.3.3
says producers `MUST NOT` generate timestamps containing leap seconds, but it
does not impose a corresponding receiver rejection rule.

The shared inbound parser therefore accepts `second = 60` consistently for
`AuthnRequest`, SSO `Response` and `Assertion`, and `LogoutResponse`. The value
is preserved rather than normalized to `59` or the following minute.

Timestamps supplied by the standard `saml-rs` generators remain limited to
seconds `00` through `59`. Applications with a separate local requirement to
reject inbound leap seconds can enforce that requirement as application policy
after parsing; it should not be described as an OASIS receiver requirement.

## Migration checklist

- Add a UTC `IssueInstant` ending in `Z` to every custom `AuthnRequest`.
- Add `request.issueInstant` to manually constructed `FlowResult` fixtures.
- Add a UTC `IssueInstant` ending in `Z` to every custom SSO `Response` and
  selected `Assertion`.
- Add `response.issueInstant` and `assertion.issueInstant` to manually
  constructed SSO `FlowResult` fixtures.
- Add a UTC `IssueInstant` ending in `Z` to every custom `LogoutResponse`.
- Add `response.issueInstant` to manually constructed LogoutResponse
  `FlowResult` fixtures.
- Update custom LogoutResponse templates to emit the required root attributes,
  correlated `InResponseTo`, entity-format `Issuer`, and final `Status` in
  schema order.
- Remove root `ds:Signature` elements from custom LogoutResponse templates.
- Remove calls to `RespondSlo::signing(...)`; typed responses are always
  signed.
- Configure a usable signing key and certificate for every typed SLO
  responder.
- Update local policy or tests that previously expected inbound leap seconds
  to be rejected.
- Update tests or telemetry that expect another error for malformed unsigned
  requests or responses.
- No change is needed for conforming messages generated by the standard
  `saml-rs` APIs.
