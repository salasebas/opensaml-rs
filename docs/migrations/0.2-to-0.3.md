# Migrating from 0.2 to 0.3

This guide covers the source changes required when upgrading from the latest
`0.2` release to `0.3`. For the complete list of changes, see the
[changelog](../../CHANGELOG.md).

## Response signature policy rename

The prerelease typed policy now names the exact XML element it governs.

Before:

```rust
use saml_rs::{MessageSignaturePolicy, SpValidationPolicy};

let mut validation = SpValidationPolicy::strict();
validation.messages = MessageSignaturePolicy::RequireSigned;
```

After:

```rust
use saml_rs::{ResponseSignaturePolicy, SpValidationPolicy};

let mut validation = SpValidationPolicy::strict();
validation.responses = ResponseSignaturePolicy::RequireSigned;
```

`MessageSignaturePolicy` and `SpValidationPolicy::messages` have been removed;
there are no deprecated aliases. The replacement variants are:

- `RequireSigned`, which authenticates every Response;
- `RequireForEncryptedCbc`, which follows the Approved Errata 05 E93
  recommendation for CBC-encrypted assertions; and
- `AllowUnsignedEncryptedCbcForCompatibility`, which explicitly relaxes that
  recommendation and is the standalone enum default used by compatibility
  configuration.

## Strict Web SSO behavior

SAML V2.0
[Approved Errata 05](https://docs.oasis-open.org/security/saml/v2.0/errata05/os/saml-v2.0-errata05-os.html)
E26 and E93 amend the Web Browser SSO HTTP-POST rules: each Assertion must be
protected by a digital signature, which can be achieved by signing each
Assertion or by signing the enclosing Response.

`SpValidationPolicy::strict()` continues to require direct trusted signature
coverage of the consumed Assertion. That is a deliberate library-hardening
policy, not the profile's only conforming signature arrangement. It also aligns
with the separate direct-Assertion requirement advertised by
`WantAssertionsSigned` in
[OASIS SAML Metadata 2.0 OS](https://docs.oasis-open.org/security/saml/v2.0/saml-metadata-2.0-os.pdf),
§2.4.4. Standard SP metadata has no corresponding attribute that requires a
top-level Response signature for every plaintext response.

Strict policy uses `ResponseSignaturePolicy::RequireForEncryptedCbc`, following
Errata 05 E93's recommendation to sign a Response containing a CBC-encrypted
Assertion so the ciphertext has integrity protection. Plaintext responses do
not require Response-root coverage unless the deployment opts in explicitly:

```rust
use saml_rs::{ResponseSignaturePolicy, SpValidationPolicy};

let validation = SpValidationPolicy {
    responses: ResponseSignaturePolicy::RequireSigned,
    ..SpValidationPolicy::strict()
};
```

The requirement is satisfied only by trusted signature evidence covering the
Response root. An Assertion-only signature does not satisfy it. Verified
detached HTTP-Redirect signatures and HTTP-POST-SimpleSign CD04 signatures
continue to satisfy the response-message requirement within those supported
bindings.

When any of these binding-level messages is authenticated, the applicable
binding requires `Response@Destination`; the typed receiver now rejects a
missing value as well as a mismatch. An unsigned HTTP-POST Response whose
Assertion is signed directly does not acquire that conditional binding
requirement.

## Signing typed IdP Responses

Typed IdPs automatically sign the top-level Response when they generate an
`EncryptedAssertion` with a known CBC-mode algorithm. Use the per-response
option to force an embedded top-level XML signature for any HTTP-POST response:

```rust
let options = RespondSso::post().sign_response();
let response = idp.respond_sso(sp, &request, subject, options)?;
```

For HTTP-POST-SimpleSign, detached signing remains binding-defined;
`sign_response()` does not replace it with an embedded XML signature.

To relax the CBC recommendation for a legacy peer, opt out explicitly:

```rust
let options =
    RespondSso::post().allow_unsigned_encrypted_cbc_for_compatibility();
let response = idp.respond_sso(sp, &request, subject, options)?;
```

## Public time types now come from `std`

The public validation and replay APIs now use [`std::time::SystemTime`] and
[`std::time::Duration`] instead of types from the `time` crate. SAML timestamp
parsing and comparison still use `time` internally.

The affected public APIs are:

- `SamlValidationContext::new` and `SamlValidationContext::now`;
- `SamlValidationContext::with_replay_retention` and
  `SamlValidationContext::replay_retention`;
- `ReplayCache::check_and_store`;
- `FlowOptions::now` in the raw compatibility API.

### Update validation clocks and replay retention

Replace `OffsetDateTime::now_utc()` with `SystemTime::now()`, and replace
`time::Duration` constructors with their `std::time::Duration` equivalents.

Before:

```rust
use saml_rs::{ReplayPolicy, SamlValidationContext};
use time::{Duration, OffsetDateTime};

let validation = SamlValidationContext::new(
    OffsetDateTime::now_utc(),
    ReplayPolicy::DisabledForCompatibility,
)
.with_replay_retention(Duration::minutes(5));
```

After:

```rust
use std::time::{Duration, SystemTime};

use saml_rs::{ReplayPolicy, SamlValidationContext};

let validation = SamlValidationContext::new(
    SystemTime::now(),
    ReplayPolicy::DisabledForCompatibility,
)
.with_replay_retention(Duration::from_secs(5 * 60));
```

Fixed clocks used by tests can be constructed relative to
`SystemTime::UNIX_EPOCH` with `checked_add` or `checked_sub`.

### Update replay cache implementations

`ReplayCache` implementations must accept a `SystemTime` expiration value.

Before:

```rust
use saml_rs::{ReplayCache, ReplayKey, SamlError};
use time::OffsetDateTime;

struct Cache;

impl ReplayCache for Cache {
    fn check_and_store(
        &mut self,
        key: ReplayKey,
        expires_at: OffsetDateTime,
    ) -> Result<(), SamlError> {
        // Store the key and expiration atomically.
        let _ = (key, expires_at);
        Ok(())
    }
}
```

After:

```rust
use std::time::SystemTime;

use saml_rs::{ReplayCache, ReplayKey, SamlError};

struct Cache;

impl ReplayCache for Cache {
    fn check_and_store(
        &mut self,
        key: ReplayKey,
        expires_at: SystemTime,
    ) -> Result<(), SamlError> {
        // Store the key and expiration atomically.
        let _ = (key, expires_at);
        Ok(())
    }
}
```

If the cache backend uses another timestamp representation, convert at the
storage boundary. Preserve subsecond precision and instants before the Unix
epoch when the backend supports them.

### Update raw `FlowOptions`

Callers that set a deterministic validation instant through the raw API must
now assign a `SystemTime`:

```rust
use std::time::SystemTime;

use saml_rs::raw::FlowOptions;

let mut options = FlowOptions::default();
options.now = Some(SystemTime::now());
```

## Error behavior

Conversions between `SystemTime` and the internal SAML timestamp range are now
checked. An unsupported validation instant returns `SamlError` instead of
panicking. An invalid or unrepresentable replay expiration returns
`SamlError::TimeWindowInvalid` with
`TimeWindowField::ReplayExpiration`.

Zero replay retention and retention values that overflow `SystemTime` also fail
closed with the replay-expiration error.

## Dependency cleanup

Applications that depended directly on `time` only to construct validation
contexts, configure replay retention, or implement `ReplayCache` can remove that
dependency after migrating all call sites. Applications may continue using
`time` for their own purposes; it is no longer required by these public API
contracts.

This migration does not change feature flags or the minimum supported Rust
version.

## Checklist

- Replace public `OffsetDateTime` validation values with `SystemTime`.
- Replace public `time::Duration` replay values with `std::time::Duration`.
- Update every `ReplayCache::check_and_store` implementation.
- Update raw `FlowOptions::now` assignments.
- Remove a direct `time` dependency if no application code still uses it.
- Run validation and replay tests, including fixed-clock and expiration cases.
