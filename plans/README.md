## Dependency notes

- 011 is the decision point: `Saml` is the public typed facade, and
  `FlowResult` moves to raw compatibility. Do not start 012-016 before that API
  boundary is agreed.
- 012 and 013 split the foundation: typed configs/policies first, then typed
  browser/domain models. They should not modify crypto behavior.
- 017 can run after 011 and should land before 014 if possible, so the typed
  facade exposes branchable semantic errors rather than freezing legacy
  samlify-style string codes into the new API.
- 018 should run after 012 and before 013/014. Its narrowed binding and endpoint
  types prevent illegal Web Browser SSO response bindings from leaking into the
  typed facade.
- 020 should land before 014 if possible. Without caller-owned clock and replay
  policy, the typed facade would hide security-critical application state behind
  a clean method name.
- 014 is the core Web SSO user journey and should land before docs are rewritten
  around the typed API. It should consume the binding/endpoint types from 018,
  semantic errors from 017, and validation context from 020.
- 015 brings SLO into the same typed style, but it can be deferred if SSO must
  ship first. If 015 is deferred, 016 must not claim typed SLO support.
- 019 can run after 011 and in parallel with implementation, but it should land
  before 016 so docs.rs and README point at the same architecture decisions.
- 021 can run after 019 records the architecture boundary. It turns signed
  metadata trust into API shape so parsed metadata and verified peers are not
  confused in typed call sites.
- 016 is docs/examples after the typed API exists. It should include the
  unsupported-profile note for Artifact, SOAP, ECP, SAML queries, NameID
  management, and metadata federation.

## Findings considered and rejected

- Rewriting XML-DSig/XML-Enc or canonicalization in-tree: rejected. The repo
  guide requires XML crypto to stay delegated to `bergshamra`.
- Making `FlowResult` the primary typed API: rejected. It should remain raw
  compatibility because string-keyed extraction is the core ergonomics problem.
- Full generic typestate over role, flow, message, binding, and verification:
  rejected for now. `Saml<Sp>` / `Saml<Idp>` plus typed transaction tokens gives
  useful safety without heavy monomorphization or unreadable docs.rs pages.
- A single enum-driven `exchange(SpOp)` facade: rejected as the primary shape.
  It is compact, but method names are easier to discover in rustdoc.
- Adding HTTP framework, async trait, serde, or native C dependencies as part
  of the typed API: rejected. Keep the facade framework-neutral and compile-time
  cheap unless a separate plan proposes otherwise.
- Copying `danielkov/saml` crypto, federation, or proxy internals: rejected.
  The useful borrow is API structure: role-typed config, explicit dispatch,
  replay/clock inputs, semantic errors, and verified metadata boundaries.
